# RocketCTL - Full Project Overview

This document is the single source of truth for building the RocketCTL CLI tool.

---

## What is RocketCTL

A convention-based CLI tool that orchestrates Docker image building, versioning, pushing, and deployment for any project. It works by reading a minimal `rocket.yaml` config file and following folder structure conventions.

The user navigates to any project directory, and RocketCTL handles the rest.

---

## Design Principles

1. Convention over configuration. Minimal config, infer the rest from folder structure.
2. Project-local. The user `cd`s into a project directory and runs commands there.
3. Works for any Docker-based project. Not tied to any specific tech stack.
4. Supports both monorepo (multiple services in one repo) and single-service repos.
5. MVP-first. Simple, fast to build, easy to extend later.

---

## How It Works

RocketCTL expects a `rocket.yaml` file in the project root. This file provides the minimal information that cannot be inferred: project name, registry URL, and service list.

Everything else follows conventions:

- Each service lives in a subfolder matching its name (monorepo) or in the root (single-service repo).
- Dockerfiles follow a naming convention: `Dockerfile` for dev, `Dockerfile.production` for prod, `.dockerignore` for ignoring files.
- Environment files follow a naming convention: `.env`, `.env.development`, `.env.production`.
- Version tracking uses `.rocket-version` files (one per service, managed by RocketCTL).

## Configuration File: rocket.yaml

### Single-Service Repo

When a repo contains only one service (e.g., a standalone backend):

```yaml
project: my-new-project-backend
service: backend
registry: 423756184128.dkr.ecr.us-east-2.amazonaws.com
region: us-east-2
domain: api.myapp.com
```

Fields:

- `project` (required): Used as image name prefix. Images are named `<project>_<service>`.
- `service` (required for single-service repos): The service name.
- `registry` (required): Container registry URL.
- `region` (required for AWS ECR): AWS region.
- `domain` (optional): Used when generating Caddy templates.

### Monorepo

When a repo contains multiple services:

```yaml
project: cochinitoapp
registry: 423756184128.dkr.ecr.us-east-2.amazonaws.com
region: us-east-2
domain: cochinito.app

services:
  - api
  - web
```

Fields:

- `project` (required): Same as above.
- `services` (required for monorepo): Explicit list of service folder names.
- `registry` (required): Same as above.
- `region` (required for AWS ECR): Same as above.
- `domain` (optional): Same as above.

### Determining Repo Type

RocketCTL determines the mode based on the config:

- If `service` field is present: single-service mode. Dockerfiles live in the project root.
- If `services` field is present: monorepo mode. Each service name maps to a subfolder.
- Both cannot be present at the same time. Validation should reject this.

## Folder Structure Conventions

### Monorepo Layout

```
~/my-project/
  rocket.yaml
  docker-compose.yml            # Dev environment (user-created, user-maintained)
  docker-compose.prod.yml       # Production (generated by init, user customizes)
  caddy/
    Caddyfile                   # Reverse proxy (generated template, user customizes)
  api/
    Dockerfile                  # Dev build
    Dockerfile.production       # Production build
    .dockerignore
    .env                        # Fallback environment
    .env.development            # Dev-specific environment
    .env.production             # Prod-specific environment (not in git)
    .env.production.example     # Template for .env.production (in git)
    .rocket-version             # Managed by RocketCTL
    [application code]
  web/
    Dockerfile
    Dockerfile.production
    .dockerignore
    .env
    .env.development
    .env.production
    .env.production.example
    .rocket-version
    [application code]
```

### Single-Service Layout

```
~/my-project-backend/
  rocket.yaml
  docker-compose.yml
  docker-compose.prod.yml
  Dockerfile                    # Dev build (in root, not subfolder)
  Dockerfile.production         # Production build (in root)
  .dockerignore
  .env
  .env.development
  .env.production
  .env.production.example
  .rocket-version
  [application code]
```

### Key Conventions

- `Dockerfile` is used for development (may include hot-reload, dev dependencies).
- `Dockerfile.production` is used for production builds (optimized, no dev tools).
- `.dockerignore` applies to both Dockerfiles.
- `.rocket-version` contains a single semver string (e.g., `0.1.0`). Managed entirely by RocketCTL. Auto-created with `0.1.0` if missing.
- `.env.production` must never be committed to git. Use `.env.production.example` as a template.

## Image Naming Convention

Images follow the pattern: `<project>_<service>:<version>`

Examples:

- Project `cochinitoapp`, service `api` -> `cochinitoapp_api:0.1.0`
- Project `cochinitoapp`, service `web` -> `cochinitoapp_web:0.2.0`
- Project `my-new-project-backend`, service `backend` -> `my-new-project-backend_backend:0.1.0`

Full registry path: `<registry>/<project>_<service>:<version>`

Example: `423756184128.dkr.ecr.us-east-2.amazonaws.com/cochinitoapp_api:0.1.0`

## Version Management

### Storage

Each service has a `.rocket-version` file containing a single semver string.

- Monorepo: `./api/.rocket-version`, `./web/.rocket-version`
- Single-service: `./.rocket-version`

### Bump Types

- `patch`: 0.1.0 -> 0.1.1 (default)
- `minor`: 0.1.0 -> 0.2.0
- `major`: 0.1.0 -> 1.0.0

### Version Lifecycle

1. `rocketctl build` reads the current version, bumps it, builds the image, and writes the new version only on success.
2. `rocketctl test` builds the image using the current version without bumping.
3. The `.rocket-version` file is the source of truth for what has been built locally.

## Environment Variables

### Lookup Convention

RocketCTL recognizes these environment files per service directory (or project root for single-service):

- `.env` - Shared/fallback values
- `.env.development` - Development-specific values
- `.env.production` - Production-specific values (secrets, not committed to git)
- `.env.production.example` - Template showing required production variables (committed to git)

### Usage by Commands

- `rocketctl dev`: Uses the existing `docker-compose.yml` which references whatever env files the user configured.
- `rocketctl build`: If `.env.production` exists in the service directory, loads each key-value pair and passes them as `--build-arg` to Docker. This is needed for frameworks like Next.js that bake `NEXT_PUBLIC_*` variables into the build.
- `rocketctl test`: Same as build for env handling.
- Runtime containers: Use `env_file` directives in docker-compose files to load the appropriate `.env` file.

## Commands

### rocketctl init

Initializes a project for use with RocketCTL.

Behavior:

1. Prompts for: project name (defaults to current directory name), registry URL, AWS region, domain (optional).
2. Asks if this is a monorepo or single-service repo.
3. For monorepo: asks for service names (comma-separated).
4. For single-service: asks for service name.
5. Creates `rocket.yaml`.
6. Creates `.rocket-version` file(s) initialized to `0.1.0`.
7. Generates `docker-compose.prod.yml` template (user expected to customize).
8. Generates `caddy/Caddyfile` template if domain was provided (user expected to customize).
9. Generates `.env.production.example` template(s).

Does NOT generate Dockerfiles. Those are project-specific and must be created by the user.

### rocketctl build [service] --bump [patch|minor|major]

Builds a production Docker image for a service.

Behavior:

1. Reads `rocket.yaml` to determine project name and registry.
2. Resolves the service directory: subfolder `./[service]/` for monorepo, project root `./` for single-service.
3. Validates the service name exists in the config.
4. Reads `.rocket-version` from the service directory.
5. Bumps version according to `--bump` flag (default: `patch`).
6. If `.env.production` exists in the service directory, loads it and passes each key-value pair as `--build-arg` to Docker.
7. Runs: `docker build --no-cache -t <project>_<service>:<new_version> -t <registry>/<project>_<service>:<new_version> -f <service_dir>/Dockerfile.production <service_dir>`
8. On success: writes the new version to `.rocket-version`.
9. On failure: does not update the version file. Prints error.

For monorepo: `rocketctl build api --bump patch`
For single-service: `rocketctl build --bump patch` (service name inferred from config)

### rocketctl push [service]

Pushes a built image to the container registry.

Behavior:

1. Reads `rocket.yaml` for registry info.
2. Resolves and validates the service.
3. Reads `.rocket-version` for current version.
4. Authenticates with the registry (for AWS ECR: runs `aws ecr get-login-password --region <region>` piped to `docker login --username AWS --password-stdin <registry>`).
5. Runs: `docker push <registry>/<project>_<service>:<version>`

For monorepo: `rocketctl push api`
For single-service: `rocketctl push`

### rocketctl dev [service]

Starts the development environment using docker compose.

Behavior:

1. Looks for `docker-compose.yml` in the project root.
2. If no service argument: runs `docker compose up` (starts all services).
3. If service argument: runs `docker compose up <project>-<service>` (starts only that service).

Flags:

- `--build`: Rebuild images before starting.
- `--no-cache`: Rebuild without cache (only valid with `--build`).

The user is responsible for creating and maintaining `docker-compose.yml` for their dev environment.

### rocketctl down

Stops the development environment.

Behavior:

1. Runs: `docker compose -f docker-compose.yml down`

### rocketctl test [service]

Tests a production build locally before pushing.

Behavior:

1. Resolves the service directory.
2. Reads `.rocket-version` for the current version (does NOT bump).
3. Builds using `Dockerfile.production` (same build logic as `build` but no version increment).
4. Runs the container in detached mode: `docker run -d --rm --env-file <service_dir>/.env.production -p <mapped_ports> <project>_<service>:<version>`
5. Prints the container ID so the user can inspect and stop it.

The user verifies the service works, then stops it manually or runs `rocketctl down`.

### rocketctl deploy

Deploys services on a production server.

Behavior:

1. Reads `rocket.yaml` for registry info.
2. Authenticates with the registry (same as push).
3. Runs: `docker compose -f docker-compose.prod.yml pull`
4. Runs: `docker compose -f docker-compose.prod.yml up -d`

This command is intended to be run on the production server where `docker-compose.prod.yml` and `.env.production` files are already in place.

### rocketctl ps

Lists running containers for the current project.

Behavior:

1. Reads `rocket.yaml` for the project name.
2. Runs: `docker ps --filter "name=<project>"` to show only containers matching the project.

### rocketctl logs [service] [-f]

Shows logs for a service.

Behavior:

1. Runs: `docker compose -f docker-compose.yml logs [-f] <project>-<service>`
2. The `-f` flag follows/tails the log output.

### rocketctl exec [service] [command...]

Executes a command inside a running container.

Behavior:

1. Runs: `docker exec -it <project>-<service> <command...>`

Example: `rocketctl exec api bash`

### rocketctl prune

Cleans up old Docker images for the current project.

Behavior:

1. Reads `rocket.yaml` for the project name.
2. Lists all local images matching the pattern `<project>_*`.
3. Identifies images that are NOT the current version (per `.rocket-version`).
4. Prints the list and asks for confirmation.
5. Removes confirmed images.

### rocketctl list

Lists all services for the current project.

Behavior:

1. Reads `rocket.yaml`.
2. For each service, reads `.rocket-version`.
3. Displays a table: service name, current version, service directory path.

### rocketctl version [service]

Shows the current version of a service.

Behavior:

1. If service specified: reads `.rocket-version` for that service.
2. If no service specified: reads `.rocket-version` for all services.
3. Displays: service name and version.

## Typical Workflows

### First-Time Setup (New Project)

```
cd ~/my-new-project
rocketctl init
# Answer prompts: project name, registry, region, domain, services

# rocketctl creates:
#   rocket.yaml
#   .rocket-version files
#   docker-compose.prod.yml (template)
#   caddy/Caddyfile (template)
#   .env.production.example files

# User then creates manually:
#   Dockerfile and Dockerfile.production per service
#   docker-compose.yml for dev environment
#   Copies .env.production.example -> .env.production and fills in secrets
#   Customizes docker-compose.prod.yml and Caddyfile as needed
```

### Local Development

```
cd ~/my-new-project
rocketctl dev                    # Start dev environment (docker compose up)
rocketctl logs api -f            # Tail API logs
rocketctl exec api bash          # Shell into API container
rocketctl down                   # Stop everything
```

### Build, Test, and Push

```
cd ~/my-new-project
rocketctl test api               # Test production build locally (no version bump)
# Verify it works...
rocketctl build api --bump patch # Build with version bump (0.1.0 -> 0.1.1)
rocketctl push api               # Push to AWS ECR
```

### Deploy to Production Server

```
# On the production server:
cd ~/production/my-new-project
rocketctl deploy                 # Authenticates, pulls latest images, restarts
rocketctl ps                     # Verify containers are running
rocketctl logs api -f            # Check for errors
```

### Maintenance

```
cd ~/my-new-project
rocketctl list                   # See all services and versions
rocketctl version api            # Check API version
rocketctl prune                  # Clean up old images
```

## Generated Templates

### docker-compose.prod.yml

Generated by `rocketctl init`. The user is expected to customize it (add databases, volumes, health checks, etc.).

Template produces a compose file with:

- Each service referencing its registry image with a version variable.
- Caddy service for reverse proxy (ports 80 and 443).
- A shared network named `<project>-network`.
- Log rotation (json-file driver, 10MB max, 3 files).
- Restart policy: `unless-stopped`.
- `env_file` pointing to the service's `.env.production`.

### caddy/Caddyfile

Generated by `rocketctl init` only if a domain is provided. The user is expected to customize it.

Template produces a Caddyfile with:

- Global email block (prompts user for email during init).
- A reverse_proxy entry per service with a domain mapping.

### .env.production.example

Generated by `rocketctl init` per service directory. Contains placeholder comments the user fills in.

## AWS ECR Prerequisites

ECR repositories must be created manually before the first push. RocketCTL does not auto-create them.

To create a repository:

```
aws ecr create-repository --repository-name <project>_<service> --region <region>
```

Example for project `cochinitoapp` with services `api` and `web`:

```
aws ecr create-repository --repository-name cochinitoapp_api --region us-east-2
aws ecr create-repository --repository-name cochinitoapp_web --region us-east-2
```

The repository URI will be: `<registry>/<project>_<service>`

ECR authentication expires after 12 hours. RocketCTL re-authenticates automatically before each push and deploy command.

AWS CLI must be installed and configured on the machine (`aws configure` or environment variables).

## Reference: automatedhub_cli

RocketCTL replaces and generalizes `automatedhub_cli` (located at `~/Documents/projects/automatedhub/automatedhub_cli`).

### What to reuse

- **Version management** (`internal/version/version.go`): The `CalculateBump` function is solid. Needs path refactoring to use `.rocket-version` instead of `automatedhub_<service>/.version`, and the service name should not be embedded in the path logic.
- **Docker operations** (`internal/docker/docker.go`): The Build, Tag, Push, and ECR login flow. Needs refactoring to accept config values instead of hardcoded constants (`host`, `servicePrefix`, per-service port maps).
- **Docker Compose wrapper** (`internal/dockerCompose/dockerCompose.go`): The compose runner. Should use `docker compose` (v2 plugin syntax) instead of `docker-compose` (v1 standalone binary).
- **Command structure** (`cmd/`): The commands build, push, test, prune, up, down, logs, ps, exec all have the right shape. They need refactoring to use config-driven service resolution.

### What NOT to reuse

- `internal/mqttClient/`: Project-specific MQTT client. Not relevant.
- `cmd/api.go`: Django-specific management command. Not relevant.
- `cmd/mqttClient.go`: MQTT-specific command. Not relevant.

### Known bugs to fix in the rewrite

- `cmd/up.go`: Flags `--build` and `--no-cache` are registered on `buildCmd` instead of `upCmd`. Must register on the correct command.
- `version.MarkPushed()` overwrites the `.pushed` file instead of appending. For the MVP, RocketCTL will not track pushed versions separately. The prune command will compare local images against the current `.rocket-version` instead.
- `version.Bump()` and `docker.Stop()` are dead code. Do not carry over.
- The `--env` persistent flag on root command is declared but never used. Do not carry over.
- No service name validation. RocketCTL must validate that the service argument matches the config.
- ECR host is hardcoded in multiple places. RocketCTL reads it from `rocket.yaml`.
- `docker.Build` special-cases the `web` service for loading `.production.env` as build-args. RocketCTL applies this generically: if `.env.production` exists in any service directory, load it as build-args.

## Technology Stack

- Language: Go (1.23+)
- CLI framework: github.com/spf13/cobra
- Config parsing: gopkg.in/yaml.v3
- Container runtime: Docker and Docker Compose v2 (`docker compose` not `docker-compose`)
- Container registry: AWS ECR
- Reverse proxy: Caddy (for generated templates only)

## Scope Boundaries

### In Scope (MVP)

- `rocket.yaml` config file (minimal, convention-based)
- All commands listed above: init, build, push, dev, down, test, deploy, ps, logs, exec, prune, list, version
- Monorepo and single-service repo support
- AWS ECR authentication and push
- Semantic versioning per service
- Template generation for docker-compose.prod.yml, Caddyfile, .env.production.example
- Environment file conventions (.env, .env.development, .env.production)
- Service name validation against config

### Out of Scope (Not MVP)

- Remote deployment via SSH
- CI/CD integration
- Rollback mechanisms
- Health check automation
- Log aggregation
- Multi-registry support (only AWS ECR)
- Automatic Dockerfile generation
- Kubernetes support
- Multi-environment beyond dev/production
- Auto-creating ECR repositories (manual, documented)
